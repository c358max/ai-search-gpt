# 13. restore index TODO

## 0. 목적
- 현재 인덱스 롤아웃은 alias 전환 직후 이전 인덱스를 즉시 삭제한다.
- 이 구조는 배포 직후 검색 이상이 발생했을 때 빠른 rollback 여지를 거의 남기지 않는다.
- 상용화 전에는 최소한 최근 버전 인덱스를 N개 보관해서 alias 복구만으로 이전 버전으로 되돌릴 수 있어야 한다.

## 1. 권장 구현

### 1.1 설정 추가
- `ai-search.index-retention-count=3`

### 1.2 cleanup 정책 변경
- 현재:
  - `oldIndex` 1개 바로 삭제
- 변경:
  - `indexName` prefix로 버전 인덱스 목록 조회
  - alias 대상 인덱스 제외
  - 최신 N개 유지
  - 나머지 삭제

### 1.3 정렬 기준
- 가장 쉬운 건 인덱스명 정렬
- 현재 버전 인덱스가 timestamp suffix면 문자열 정렬로 충분

### 1.4 retention 기준 확정
- `N개 보관`의 의미:
  - **현재 `readAlias`가 가리키는 인덱스를 포함한 전체 N개 보관**
- retention 대상:
  - **`${indexName}-v*` 패턴의 버전 인덱스만 대상**
- alias 미존재 시 정책:
  - **정상 수행**
  - 버전 인덱스 목록 기준으로 최신 N개만 유지
- 최소 보관 개수:
  - **`index-retention-count`는 최소 2 이상 강제**

## 2. 예상 코드 변경 범위
- `AiSearchProperties.java`
  - retention count 추가
- `application.yml`
  - 기본값 추가
- `IndexCleanupService.java`
  - 단건 삭제 -> retention cleanup 로직 변경
- `IndexRolloutService.java`
  - alias 전환 후 `cleanupOldVersionedIndices()` 호출

## 3. 권장 기본값
- 개발: `2`
- 상용: `3` 또는 `5`

## 4. 가장 단순한 운영 규칙
- 현재 `readAlias` 인덱스는 절대 삭제 안 함
- 최신 버전 인덱스 N개 보관
- 그 외 버전 인덱스만 삭제
- 삭제 전 삭제 대상 목록을 로그로 남김
- retention 대상은 `${indexName}-v*` 패턴만 허용

## 5. 테스트 케이스

### 5.1 인덱스가 없거나 N개 이하인 경우
- 현재 인덱스 개수 출력
- 삭제 대상 없음 출력
- 삭제 실행 안 함

### 5.2 인덱스가 N개 초과인 경우
- 삭제 대상 인덱스 목록 출력
- 실제 삭제된 인덱스명 출력
- 삭제 개수 1 이상 검증

### 5.3 공통 검증
- 현재 `readAlias` 인덱스는 삭제되지 않아야 함
- `${indexName}-v*` 외 인덱스는 cleanup 대상이 아니어야 함
- retention count가 2 미만이면 예외 처리해야 함

## 6. 결론
- `즉시 삭제`를 `retention cleanup`으로 바꾸는 것이 가장 쉽고 안전하다.
- 별도 rollback 시스템을 새로 만드는 것보다 이 방식이 현재 코드에 가장 잘 맞는다.
